# stepflow v0.2 기능 로드맵 설계 (listeners · retry · chunk)

> 작성일: 2026-06-16
> 상태: 설계 승인 완료 (브레인스토밍), 개별 sub-spec 분해 대기
> 목표: v0.1(모듈화 완료, publish 준비) 위에 retry · listeners · chunk 세 기능을 **순수 추가적**으로 얹되, 셋의 의존·순서·공통 인터페이스를 하나의 로드맵으로 확정한다. 개별 기능은 이 문서 아래 sub-spec → plan → 구현 사이클로 진행한다.

## 0. 전제 (v0.1 현황)

- `@stepflow/core`는 스텝 단위 플로우 러너: `defineJob().step().branch().build()` + `runJob()`. 실패 스텝부터 restart, `shared` 컨텍스트(스텝 경계 스냅샷/복원), exit-status 기반 분기.
- 동봉 repo 2종: `InMemoryJobRepository`(core), `MySqlJobRepository`(infrastructure, mysql2 peer). 공용 계약 스위트 `@stepflow/test`.
- 0.1은 publish 준비 완료(메타데이터·dual빌드·dry-run green). v0.2는 SemVer **minor**로 배포 가능해야 한다.

## 1. 확정된 결정 (브레인스토밍)

| 항목        | 결정                                                                                                                                                                        |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 호환성 원칙 | **순수 추가적** — `defineJob`/`runJob`/`JobRepository` 기존 시그니처 불변. 옵션·신규 API로만 확장. 0.1 사용자 코드 그대로 동작                                              |
| 영속화 수준 | **영속화 + repo 선택적 확장** — retry attempt·chunk 체크포인트를 메타데이터에 저장. 신규 repo 메서드는 **optional**, 엔진이 존재 확인 후 호출, 없으면 graceful degrade      |
| 레이어링    | **C 하이브리드** — listeners를 라이프사이클 이벤트 substrate로 두되, retry/chunk는 **단독 동작**하고 listeners가 등록돼 있으면 자기 이벤트를 그 버스로 emit. 강제 결합 없음 |
| 구현 순서   | listeners → retry → chunk (각각 독립 sub-spec). retry를 먼저 빼도 무방하나, listeners 우선 시 retry/chunk가 처음부터 이벤트 emit하도록 작성 가능                            |
| 버전        | 전체 **0.2.0 (minor)**. 코어 공개 API breaking 0건                                                                                                                          |
| 리뷰 관행   | 각 단계 완료 시 기존 게이트(typecheck/lint/test/dual빌드/grep) green 유지 + codex 비판 리뷰(v1 스펙 체크포인트 관행 계승)                                                   |

## 2. 아키텍처 & 레이어링 (C 하이브리드)

```
                ┌─────────────────────────────────────┐
                │  Lifecycle Event Bus (listeners)     │  ← 선택적 substrate
                │  beforeJob / afterJob                │
                │  beforeStep / afterStep / onStepError│
                │  onRetry          (retry가 emit)     │
                │  onChunk          (chunk가 emit)     │
                └─────────────────────────────────────┘
                     ▲              ▲             ▲
              runJob 코어      retry 정책      chunk 스텝
              (이벤트 방출)   (스텝 래핑)    (새 스텝 종류)
```

의존 관계 (단방향, 강제 결합 없음):

| 기능          | 의존                | 단독 동작                | 영속화                        |
| ------------- | ------------------- | ------------------------ | ----------------------------- |
| **listeners** | 코어 엔진만         | —                        | 안 함 (순수 콜백)             |
| **retry**     | 엔진 스텝 실행 루프 | ✅ listeners 없어도 동작 | attempt 횟수 (repo 선택적)    |
| **chunk**     | 엔진 + 영속화       | ✅                       | 청크 체크포인트 (repo 선택적) |

핵심 불변식 (전 기능 공통):

- `runJob`/`defineJob` 기존 시그니처 불변 — 전부 옵션 추가로만.
- `JobRepository`에 추가되는 **메서드는 전부 optional** → 엔진이 `repository.xxx?` 존재 확인 후 호출, 없으면 graceful degrade(영속화 생략, 기능 자체는 in-memory로 계속 동작).
- 동봉 repo 2종(InMemory/MySql)은 신규 메서드 구현. 커스텀 repo는 안 깨짐.

## 3. listeners (substrate)

### 인터페이스 (core/types.ts에 추가)

```ts
export interface JobListener {
  beforeJob?(ctx: JobLifecycleContext): void | Promise<void>;
  afterJob?(ctx: JobLifecycleContext, result: RunJobResult): void | Promise<void>;
  beforeStep?(ctx: StepContext): void | Promise<void>;
  afterStep?(ctx: StepContext, outcome: StepOutcome): void | Promise<void>;
  onStepError?(ctx: StepContext, error: unknown): void | Promise<void>;
  onRetry?(ctx: StepContext, info: RetryInfo): void | Promise<void>; // retry sub-spec에서 활성
  onChunk?(ctx: StepContext, info: ChunkInfo): void | Promise<void>; // chunk sub-spec에서 활성
}

export interface StepOutcome {
  readonly status: 'COMPLETED' | 'FAILED';
  readonly exitStatus: ExitStatus;
  readonly durationMs: number;
  readonly error?: string;
}
```

`runJob` 옵션에 `readonly listeners?: readonly JobListener[]` 추가.

### 방출 시점 (run-job.ts 기존 루프에 삽입)

- `beforeJob` → 인스턴스/실행 생성 직후, 첫 스텝 전 1회
- `beforeStep` → `startStep` 후 `step.run` 전
- `afterStep` → `finishStep` 후 (성공/실패 모두)
- `onStepError` → catch 블록에서 (throw 또는 FAILED 반환 시)
- `afterJob` → `finishExecution` 후, 결과 반환 전 1회

### 핵심 결정

- **예외 격리**: listener가 throw해도 job 흐름은 중단하지 않음 — 잡아서 `logger.error`로 기록만. listener는 관측·부수효과(알림/메트릭)용이며 흐름 제어용이 아니다(흐름 제어는 step 반환값/branch가 담당).
- **순서**: 배열 등록 순서대로 순차 await(병렬 아님 — 결정적 동작·로깅 순서 보장).
- **restart 시 carry-forward된 스킵 스텝**: `beforeStep`/`afterStep`을 emit하지 않는다(실제로 실행되지 않으므로). "이벤트 = 실제 실행" 의미 일관성 유지.
- 영속화 없음 — 순수 in-process 콜백. repo 변경 없음.

### 테스트

`@stepflow/test`에 호출 순서·예외 격리 검증용 `RecordingListener` 더블 추가.

## 4. retry

### 정책 정의 (defineJob 빌더 확장 — 순수 추가)

```ts
defineJob('orders_sync')
  .step('login', loginRun)
  .step('search', searchRun)
  .retry('search', { maxAttempts: 3, backoff: { delayMs: 1000, multiplier: 2 } })
  .build();
```

```ts
export interface RetryPolicy {
  readonly maxAttempts: number; // 총 시도(최초 1 + 재시도). 미적용 시 1
  readonly backoff?: BackoffPolicy; // 없으면 즉시 재시도
  readonly retryOn?: (error: unknown) => boolean; // 기본: 모든 throw 재시도
}
export interface BackoffPolicy {
  readonly delayMs: number;
  readonly multiplier?: number; // exponential. 기본 1(고정)
  readonly maxDelayMs?: number; // 상한
}
export interface RetryInfo {
  readonly stepName: string;
  readonly attempt: number; // 방금 실패한 시도 번호(1-based)
  readonly maxAttempts: number;
  readonly error: string;
  readonly nextDelayMs: number; // 다음 시도까지 대기. 마지막 실패면 0
}
```

### 엔진 동작 (run-job.ts 스텝 실행 지점만 래핑)

- 스텝을 try/catch 루프로 감싸 `maxAttempts`까지 재시도.
- **재시도 트리거: throw만.** 명시적 `return FAILED`는 "의도된 종료 상태"이므로 재시도하지 않는다(RPA에서 throw=일시 장애, FAILED 반환=비즈니스 판단으로 구분). `retryOn`으로 throw 중에서도 필터 가능.
- 각 실패 후 `onRetry` listener emit + backoff 대기 → 마지막 시도까지 실패하면 기존과 동일하게 FAILED 처리(restart 대상).
- **백오프 대기**는 `await delay()`. `delay` 구현은 주입 가능하게(테스트 fake timer). `Date.now()`는 코어가 이미 쓰는 패턴 그대로.

### 영속화 (repo 선택적 확장, graceful degrade)

- `StepExecution`에 `attempts: number` 추가(metadata.ts). 성공 스텝도 `attempts=1`로 기록 → 관측성.
- `FinishStepInput`에 `attempts?` 추가 — **기존 메서드 입력 확장**(optional 필드). 동봉 repo 2종 반영, 커스텀 repo는 무시해도 동작.
- 중간 attempt마다 별도 row를 남기지 않고 **최종 attempts 카운트만** 기록(스키마 단순 유지). 상세 시도 이력이 필요하면 `onRetry` listener로 수집.

### restart와의 관계

retry는 단일 run 내에서 소진. maxAttempts를 다 쓰고 FAILED → 다음 `runJob` 호출 시 기존 restart 로직이 그 스텝부터 재개(attempts는 새 execution에서 0부터). 즉 retry(일시 장애 흡수)와 restart(실행 간 재개)는 직교.

### 테스트

고정/지수 백오프, throw 재시도 vs FAILED 비재시도, maxAttempts 소진 후 FAILED, `onRetry` emit 횟수, attempts 영속화 — InMemory 계약에 추가.

## 5. chunk (가장 큰 추가)

### 개념

대량 아이템을 read→process→write로 처리하되 N개마다 커밋하고 그 지점을 체크포인트로 남겨 **크래시/실패 시 마지막 커밋 청크부터 재개**. Spring Batch chunk 모델의 핵심.

### API (새 스텝 종류 — 기존 step과 공존)

```ts
defineJob('orders_sync')
  .step('login', loginRun)
  .chunkStep('process-orders', {
    chunkSize: 50,
    reader: (ctx) => AsyncIterable<Order> | Iterable<Order>, // 아이템 스트림
    processor: (item, ctx) => Promise<Result>, // 선택(없으면 item 그대로)
    writer: (results, ctx) => Promise<void>, // 청크 단위 커밋
  })
  .step('cleanup', cleanupRun)
  .build();
```

`Step`을 유니온으로 확장: `type Step = SimpleStep | ChunkStep`. 기존 `Step`은 `SimpleStep`으로 유지(추가적).

### 엔진 동작

- reader에서 아이템을 끌어와 `chunkSize`만큼 모으면 → processor 적용 → writer 호출 → **커밋 + 체크포인트 저장** → `onChunk` emit.
- 카운트 추적: `read / processed / written / committed chunks`(Spring Batch StepCounts).
- writer 성공 후에만 체크포인트 전진 → writer 실패 시 그 청크는 미커밋 상태로 FAILED.
- `onChunk` listener에 넘기는 `ChunkInfo`의 최종 형태는 chunk sub-spec에서 확정한다(대략 `{ stepName, chunkIndex, readCount, writeCount, commitCount }`).

### 영속화 / restart (repo 선택적 확장)

- 체크포인트 = "지금까지 커밋된 아이템 수(read offset)". `StepExecution`에 `readCount/writeCount/commitCount` + 청크 컨텍스트(offset) 저장.
- 신규 optional repo 메서드: `saveChunkCheckpoint?(stepExecutionId, checkpoint)` / `loadChunkCheckpoint?(...)`. **없으면 graceful degrade** → 청크 처리는 정상 동작하되 restart 시 처음부터(경고 로그 1회).
- restart 시 reader는 체크포인트 offset만큼 **스킵 후 재개**. 단 **reader가 결정적(같은 순서 재생)이어야 함** — 이 계약을 문서에 명시(비결정 reader는 중복/누락 위험).

### 핵심 결정 / 범위 제약

- **idempotent writer 권장** 명시: 크래시가 writer 커밋과 체크포인트 저장 "사이"에 나면 마지막 청크 재처리 가능(**at-least-once**). 트랜잭션 통합(writer+체크포인트 원자 커밋)은 **비범위** — 로드맵 후속.
- retry와의 결합: chunkStep에도 retry 정책 적용 가능하되 재시도 단위는 "실패한 청크"(아이템 전체 아님). v0.2에서는 단순화 위해 **chunk-level retry는 옵션이되 기본 off**, 세부는 chunk sub-spec에서 확정.
- **skip 정책**(Spring Batch faulty item skip)은 **비범위** → 후속.

### 테스트

chunkSize 경계, 부분 청크(마지막), writer 실패 후 체크포인트 미전진, restart offset 재개, repo 미지원 시 degrade, `onChunk` 카운트 — InMemory 계약 + 결정적 reader 더블.

## 6. 리포지토리 계약 변경 종합

| 변경                                                                   | 종류                      | degrade 동작                               |
| ---------------------------------------------------------------------- | ------------------------- | ------------------------------------------ |
| `FinishStepInput`에 `attempts?`, `readCount?/writeCount?/commitCount?` | 기존 입력에 optional 필드 | 무시해도 OK                                |
| `StepExecution`에 동일 필드 추가                                       | 메타데이터 확장           | 조회만, 없으면 undefined                   |
| `saveChunkCheckpoint?` / `loadChunkCheckpoint?`                        | 신규 optional 메서드      | 미구현 시 chunk restart 처음부터(경고 1회) |

- InMemory/MySql: 신규 필드·메서드 전부 구현. `schema.sql`은 `step_execution` 컬럼 추가 + (선택) `chunk_checkpoint` 저장 → 기존 0.1 스키마 사용자용 **마이그레이션 `ALTER TABLE` SQL 별도 제공**.
- listeners는 영속화 없음 → repo 변경 없음.

## 7. 단계 분해 (각각 sub-spec → plan → 구현)

1. **listeners**: `JobListener`/`StepOutcome`/`JobLifecycleContext` 타입 + `runJob` emit 지점 + 예외 격리 + `RecordingListener` 더블. repo 무변경.
2. **retry**: `RetryPolicy`/`BackoffPolicy`/`RetryInfo` + `defineJob().retry()` + 스텝 래핑 루프 + 주입형 `delay` + `attempts` 영속화 + `onRetry` emit.
3. **chunk**: `ChunkStep`/`chunkStep()` + 청크 엔진 루프 + 카운트/체크포인트 영속화 + restart offset 재개 + `onChunk` emit + 마이그레이션 SQL.

각 단계 종료 게이트: 전 패키지 typecheck·lint·test·dual빌드 green, agent-server grep 0 유지, codex 비판 리뷰 critical 0.

## 8. 성공 기준

- 세 기능 모두 **순수 추가적** — 0.1 사용자 코드 무수정 동작, 커스텀 `JobRepository` 무수정 동작(graceful degrade 검증).
- listeners: 라이프사이클 이벤트가 실제 실행 스텝에 대해서만, 등록 순서대로, 예외 격리되어 발생.
- retry: throw만 재시도·FAILED 비재시도, 백오프 동작, maxAttempts 소진 후 FAILED, attempts 영속화.
- chunk: 체크포인트 기반 restart(결정적 reader 가정), at-least-once, 카운트 영속화, repo 미지원 시 degrade.
- 전 패키지 게이트 green, codex critical 0.

## 9. 비범위 (이번 로드맵)

- writer + 체크포인트 **원자 트랜잭션**(at-least-once에 머무름)
- chunk **faulty item skip** 정책
- chunk **병렬 처리**
- **listener 기반 흐름 제어**(listener는 관측·부수효과 전용)
- 위 항목은 향후 단계로 명시적으로 남긴다.
